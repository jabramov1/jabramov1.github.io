AWS internship / architecture case study

Billing notifications as a configurable event system.

I helped design and build an event-driven notification path for AWS billing lifecycle emails, moving repeated pipeline work into configuration, routing rules, and a thin execution layer.

01 / Context

The problem was not sending email. It was making change safe.

Transfer Billing creates customer-facing notices across a billing lifecycle: initiated, scheduled, termination approaching, error states, and follow-up reminders. The old shape pushed each new email type toward its own pipeline work, which made growth slower and increased the chance of inconsistent behavior.

01

Billing responsibility changes

Account relationships can move through several states, each with different customer notification needs.

02

Customers need reliable updates

Duplicate, missing, or incorrect emails affect trust, billing clarity, and support load.

03

Teams need faster extension

The target was one reusable path where new notices could be added through validation and template config.

02 / Architecture

One async path, clear ownership at each step.

EventBridge owned filtering, enrichment, and invalid-event routing. Lambda stayed intentionally thin: validate the safety boundary, call the internal email service, and avoid taking over rendering, retry, or delivery concerns.

Lambda

Runs safety validation, builds the outbound request, and hands work to the email service.

Internal email service

Renders the right template, handles delivery retries, and sends the customer notification.

Config

JSON-defined notices

Event types, required template fields, and template IDs were defined outside the core execution code.

DLQ

Early invalid-event path

Malformed or unsupported events could be rejected before they reached the core notification logic.

Idempotency

Stable logical keys

Retries and replays map to the same intended customer notice instead of creating duplicate sends.

03 / Design choices

The engineering decision was where responsibility belonged.

The architecture avoided packing every concern into Lambda. Routing, validation boundaries, template selection, duplicate prevention, and delivery retries each had a clearer owner.

EventBridge

Route and enrich before execution

The event bus decoupled producers from delivery and attached the correct template ID before Lambda ran.

Lambda

Keep compute narrow

Lambda focused on validation and the email-service call, keeping rendering and delivery retries out of the function.

Configuration

Make new emails additive

A new notification could be represented as config: allowed event type, required fields, and template mapping.

04 / Reliability

Idempotency needed a business key, not a timestamp.

Retries can change transport metadata, and legitimate repeated emails can share similar payloads. The hard part was defining what "same notification" means for customers.

Logical uniqueness key
recipient email
billing relationship
notification type
scheduled date

The rule: one logical email per relationship, per day, per notification type, even if upstream retries or replays the event.

01

Timestamps were too unstable

Event time could change across retries, so it was not reliable enough to identify the intended notice.

02

Scheduled date made intent explicit

I recommended using a scheduled date field so repeated attempts pointed back to the same customer-facing event.

03

Keys expired after the protection window

Stored idempotency keys used a 14-day TTL to prevent duplicate sends without keeping data indefinitely.

05 / Testing and impact

The outcome was a reusable notification platform.

Testing covered validation edges, end-to-end email behavior, duplicate injection, and billing-cycle capacity. The project was approved and handed off with runbook documentation for the team to continue.
