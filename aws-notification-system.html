AWS Billing Notification Automation

I helped build a config-driven, event-based path for billing lifecycle emails. The goal was simple: make new customer notices faster to add while keeping duplicate prevention, validation, and handoff behavior predictable.

Context

The hard part was safe change.

Transfer Billing creates customer-facing notices across a billing lifecycle: initiated, scheduled, termination approaching, error states, and follow-up reminders. The old shape pushed each new email type toward repeated pipeline work.

  • Customers needed reliable billing status updates without duplicate, missing, or incorrect emails.
  • Engineers needed one reusable path instead of a new mini-pipeline for every notification type.
  • The system needed to absorb billing-cycle bursts without making upstream services wait on email delivery.
Architecture

One async path with clear ownership.

EventBridge handled routing and enrichment. Lambda stayed thin: validate the safety boundary, build the outbound request, and hand delivery to the internal email service.

  1. Producer

    Upstream billing services

    Publish lifecycle events when billing responsibility changes or a scheduled notice is due.

  2. Router

    EventBridge

    Filters supported event types, enriches valid events with template IDs, and routes invalid messages early.

  3. Executor

    Lambda

    Runs validation, creates the email-service request, and keeps rendering and delivery concerns out of compute.

  4. Delivery

    Internal email service

    Owns template rendering, delivery retries, and final customer notification delivery.

Design choices

Put responsibility where it belonged.

The important design work was deciding what Lambda should not own. Routing, invalid-event handling, template selection, duplicate prevention, and delivery retries each had a clearer owner.

EventBridge

Route and enrich before execution

The event bus decoupled producers from delivery and attached the correct template ID before Lambda ran.

Lambda

Keep compute narrow

Lambda focused on validation and the email-service call, keeping rendering and delivery retries out of the function.

Configuration

Make new emails additive

A new notification could be represented as config: allowed event type, required fields, and template mapping.

Reliability

Idempotency needed a business key, not a timestamp.

Retries can change transport metadata, and legitimate repeated emails can share similar payloads. The hard part was defining what "same notification" means for customers.

Logical uniqueness key
recipient email
billing relationship
notification type
scheduled date

The rule: one logical email per relationship, per day, per notification type, even if upstream retries or replays the event.

  • Timestamps were too unstable. Event time could change across retries, so it was not reliable enough to identify the intended notice.
  • Scheduled date made intent explicit. Repeated attempts pointed back to the same customer-facing event.
  • Keys expired after the protection window. A 14-day TTL prevented duplicate sends without storing keys indefinitely.
Testing and impact

Reusable platform, less repeated work.

Testing covered validation edges, end-to-end email behavior, duplicate injection, and billing-cycle capacity. The project was approved and handed off with runbook documentation for the team to continue.
